# How cBioPortal data is handled

cBioPortal aggregates many studies of uneven curation; its characteristic failure is a
*plausible-looking wrong value*, not a crash. This document records exactly what we take from
it, what we deliberately ignore, and every validation and normalization step in between. The
posture throughout is **treat the source as guilty until checked** — nothing is trusted that
hasn't passed an explicit rule.

Implementation: [`datasources/cbioportal.py`](../src/oncoreason/datasources/cbioportal.py) (fetch +
normalize), [`cases/qc.py`](../src/oncoreason/cases/qc.py) (validation), and
[`cases/build.py`](../src/oncoreason/cases/build.py) (sampling → cases). The QC report over the
real data lives in [`qc/`](../qc/); it is regenerated by
[`scripts/cbio_qc_report.py`](../scripts/cbio_qc_report.py) and the case set by
[`scripts/build_cases.py`](../scripts/build_cases.py).

## 1. What we use it for — and what we do not

| | |
|---|---|
| **Used for** | Real per-tumor **somatic mutation profiles** — the co-occurring alteration patterns that make cases biologically plausible instead of hand-composed. |
| **Not used** | Survival/outcome fields (confounded by indication — a causal-inference study, out of scope); clinical annotation fields beyond histology; panel-based wild-type inference; anything from restricted studies. |

We draw only from cBioPortal's most reliable layer — consensus somatic mutation calls in
curated cohorts — and ignore the layers where its quality is weakest.

## 2. Study selection (the allow-list)

Only studies on an explicit **allow-list** are ever queried; every other study is refused at
the connector (`_assert_allowed`). The shipped list is:

- `luad_tcga_pan_can_atlas_2018`, `lusc_tcga_pan_can_atlas_2018` — curated, whole-exome,
  single genome build (GRCh37), one sample per patient, consensus (MC3-lineage) somatic calls.

Deliberately **excluded**, with reasons:

- **GENIE and any `genie` study** — restricted redistribution; refused by substring guard.
- **The TCGA reprocessings** (`luad_tcga`, `luad_tcga_gdc`) — these are the *same patients* as
  the PanCancer Atlas cohort (verified: **566/566** patient overlap with `luad_tcga_gdc`,
  564 with the Firehose legacy set). Including them alongside the Atlas cohort would leak the
  same patient across a train/test boundary. One representative cohort only.
- **MSK panel cohorts** — targeted panels (230–468 genes). Not wrong, but they trigger the
  "absence ≠ wild-type" problem (§6); left out of v1 rather than half-handled.

Adding a study is a conscious act: it must be placed on the allow-list *after* checking its
build, assay type, and open-access status.

## 3. Normalization — the record shape we work in

Raw cBioPortal mutation records are immediately mapped to a stable internal shape
(`CBioPortalSource._normalize`) so no downstream code touches cBio's field quirks:

| Field | Handling | Why |
|---|---|---|
| `entrezGeneId` | kept as the **gene join key** | Symbols drift (HER2 ↔ ERBB2); the Entrez id is stable. |
| `proteinChange` | kept as-is (one-letter, no `p.`) | Already matches CIViC's convention; joinable directly. |
| `ncbiBuild` | canonicalized on use via `canonical_build` | Study metadata says `hg19` while records say `GRCh37` — **same build, two labels**; a naive equality check would flag a false mismatch. `hg19/GRCh37/b37 → GRCh37`. |
| `mutationStatus`, `validationStatus` | `.` → `None` (`normalize_missing`) | In TCGA these fields are **100% the placeholder `.`** — unusable. Somatic origin is taken from the *profile type* (a somatic mutation profile), not this field. |
| survival / treatment / outcome | **never read** | Confounded; out of scope. |

`normalize_missing` maps every cBioPortal spelling of absent (`.`, `NA`, `N/A`,
`[Not Available]`, `Unknown`, empty, `null`) to a single `None`, so "missing" is never
mistaken for a category.

## 4. QC battery (`cases/qc.py`)

Every ingest runs through `run_qc`, which produces a severity-graded report and **flags,
never silently drops**. Each check corresponds to a real anomaly found in the live data:

| Check | Severity | What it catches |
|---|---|---|
| `build_mixed` | high | More than one genome build within a study (coordinates not comparable). |
| `type_vs_protein` | medium | `mutationType` disagreeing with the proteinChange consequence — e.g. a `Frame_Shift_Del` whose proteinChange is a clean stop-gain `S46*` (no `fs`). A curation error; we do not trust `mutationType` as the consequence. |
| `vaf_zero_alt` | medium | A called mutation with `tumorAltCount == 0` (no supporting reads). |
| `duplicate_row` | low | Identical `(sample, gene, proteinChange)` listed more than once (all observed cases were splice pseudo-notation like `X126_splice`); would double-count. |
| `hypermutator` | low | A sample with ≥1000 mutations (passenger swamping). |

**Drop-vs-flag policy** (`cases/build.py`, `_DROP_CHECKS`):

- **Dropped** before aggregation: `duplicate_row` (extra occurrences only — one kept) and
  `vaf_zero_alt` records. They corrupt per-sample aggregation.
- **Kept but recorded** on the profile: `type_vs_protein`. The *variant* is real; only its
  consequence *label* is suspect, so we retain it and carry the flag rather than discard data.

### QC result on the shipped studies

Running the battery on the two allow-listed cohorts (1,871 records): **0 high, 6 medium
(`type_vs_protein`), 3 low (`duplicate_row`)** — a ~0.5% flag rate, all low-grade curation
noise, none touching the driver variants cases are built around. The high-consequence traps
(build mixing, placeholder status, cross-study contamination, panel inference) are handled by
*selection and design*, so they do not appear in the ingested data at all. See
[`qc/`](../qc/) for the dated report.

## 5. Profile sampling and patient de-duplication

`build_profiles`:

1. Fetch driver-gene mutations for each allow-listed study, run QC, apply the drop policy.
2. Group surviving records by sample.
3. **One representative sample per patient** — the *richest* profile (most alterations),
   tie-broken by sample id. Deterministic, not shuffle-dependent, and clinically sensible.
4. **Cross-study patient de-dup** — a patient already taken from one study is never taken
   again from another (the seen-patients guard). This is the structural defence against the
   TCGA-reprocessing leakage class, independent of the allow-list.
5. Cap per study; a seeded shuffle decides *which* patients when capping.

Driver-gene panel: EGFR, KRAS, ALK, ROS1, BRAF, MET, RET, ERBB2, NTRK1, PIK3CA, TP53, STK11,
KEAP1, NF1, RB1, NRAS. Restricting to drivers (not the whole exome) also keeps hypermutator
passenger noise out of the profiles.

## 6. The "absence ≠ wild-type" rule

For whole-exome cohorts (the shipped studies), a gene with no mutation call is genuinely
wild-type for point mutations. For **panel** cohorts this is false — a gene off the panel is
*untested*. The connector exposes `gene_panel_ids` / `panel_genes` precisely so that any
future "no mutation found" inference can be gated on per-sample panel membership. Until that
gate is wired, panel cohorts stay off the allow-list.

## 7. Annotation — evidence, not verdicts

Each alteration is annotated (`annotate_alteration`) by joining to the evidence sources **on
canonical identifiers, never on name strings**:

- **CIViC** (somatic, therapeutic) via gene + proteinChange; the CAID and ClinVar IDs come
  back on the record.
- **ClinVar** (germline pathogenicity) via the ClinVar IDs CIViC supplied — a canonical-id
  join, no re-spelling. The germline axis is kept **separate** from CIViC's somatic evidence
  (pooling them is a category error).

`actionable` is a screening flag (CIViC level A/B predictive *sensitivity* evidence), **not**
an ESCAT tier. Turning attached evidence into a tiered recommendation is interpretation, done
in the Phase-3 labelling step — so cases here carry `gold = None` by design. Example: ERBB2
(≡ HER2) Y772_A775dup resolves to CAID `CA135369`, CIViC level-A sensitivity to trastuzumab
deruxtecan, and a ClinVar "Likely pathogenic" germline record — the alias trap handled
correctly because the join is on ids, not the `HER2`/`ERBB2` string.

## 8. Split

`split_cases` assigns train/val/test/generalization. Because each case is already one patient
(§5), a seeded random split is **patient-disjoint by construction** — no leakage is possible.
The generalization slice is a held-out set of real profiles, matching the external-validation
framing.

## 9. Reproducibility and provenance

- **Deterministic**: fixed seed (17) in `build_cases.py`; the case set is reproducible from
  the committed script.
- **Cached**: all API responses cache under `data/public/cache/` (git-ignored, regenerable);
  re-runs are offline and identical.
- **Provenance on every case**: originating study, sample id, patient id, genome build, QC
  flags, and the attached evidence — carried in `Case.provenance`.
- **Nothing controlled or copyrighted** is committed; `data/cases/` and the caches are
  git-ignored; only code, docs, and the QC report are tracked.

## 10. Known limitations (stated, not hidden)

- Point mutations only in v1 (no CNA/fusion/expression yet — the schema supports them).
- Clinical context is histology only; stage / prior lines / performance status are honestly
  `None` for TCGA (it does not record lines/PS reliably), rather than imputed.
- `type_vs_protein`-flagged records are retained; a stricter mode could re-annotate them with
  a variant effect predictor (VEP) — a documented upgrade path, consistent with the Gate-3
  identity layer.
